Confirmed while automating SCRUM-9 (SCRUM-T16 to SCRUM-T20) against `https://demoqa.com/checkbox`:

- **Library mismatch vs. Zephyr documentation:** the live page renders its tree with the `rc-tree` library (antd Tree: classes like `.rc-tree`, `.rc-tree-treenode`, `.rc-tree-checkbox`, `.rc-tree-switcher`), NOT `react-checkbox-tree` (whose classes would be `.rct-*`, e.g. `.rct-option-expand-all`/`.rct-option-collapse-all`). **There is no "Expand all"/"Collapse all" toolbar anywhere in the live DOM** - confirmed by dumping the full page HTML and grepping for "expand"/"collapse"/aria-labels; only a per-node switcher exists that expands/collapses one node (and its subtree) at a time. A Zephyr case written against the old library's selectors (e.g. SCRUM-T20) cannot be automated as literally specified - flag it as not-automatable/pending rather than writing a test that can never pass, same principle as [[demoqa_alerts_dialog_quirks]] and [[demoqa_text_box_quirks]] ("verify real DOM before writing assertions"), but here the whole feature is absent rather than a text mismatch.
- **Locate nodes via `getByRole('treeitem', { name })`, WITHOUT `exact: true`.** A tree node's accessible name is the concatenation of its checkbox's `aria-label` ("Select Home") and its visible title ("Home"), i.e. `"Select Home Home"` - not just `"Home"`. `exact: true` against the plain label therefore never matches and the locator hangs until timeout. Non-exact (substring, case-insensitive) matching is what's needed and is safe here since node labels don't overlap as substrings of each other.
- **Never select tree nodes via `.rc-tree-treenode` + `.first()`/index.** The very first `.rc-tree-treenode` element in the DOM is a hidden phantom node rc-tree uses internally for measurement/virtualization (`aria-hidden="true"`, no `.rc-tree-checkbox`/`.rc-tree-switcher` child) - any locator chained off `.first()` on that class hangs waiting for a child that phantom node never has.
- **rc-tree unmounts collapsed children entirely** (not just CSS-hidden) - a descendant node must be expanded into the DOM via its ancestors' `.rc-tree-switcher` before it can be located or asserted on, even though checking/unchecking cascades to descendants regardless of whether they're currently expanded.
- **Tri-state checkbox semantics:** `.rc-tree-checkbox` has `aria-checked` = `"false"` | `"true"` | `"mixed"` (mixed = indeterminate/partial selection) - assert on `aria-checked` rather than CSS classes, it's more robust/semantic. Clicking a checkbox while it is `"mixed"` (indeterminate) **fully checks it** (re-selects everything under it), it does NOT uncheck it - to fully deselect a subtree you must first reach the fully-`"true"`-checked state, then click again.
- **`#result` is absent from the DOM (not just empty) when nothing is selected** - `toHaveCount(0)`, same "absent, not empty" pattern as `/text-box` and `/alerts`.
- **`#result`'s list items have no separator in the DOM.** It renders `<span>You have selected :</span>` followed by one `<span class="text-success">` per selected node id, with no whitespace/newline between the spans (visual stacking is CSS-only). A `toHaveText()` assertion against the whole `#result` container therefore concatenates all items with no separator (e.g. `"documentsdownloadscommands..."`), confirmed by a real assertion failure. Use the multi-element locator form instead: `expect(page.locator('#result .text-success')).toHaveText([...orderedItems])`.
- **Zephyr text vs. real site:** SCRUM-T16 documents the `#result` listing as shown "en negrita" (bold), but the real `.text-success` class has `font-weight: 400` (not bold), confirmed via `getComputedStyle`.
- **`#result` item order is fixed and does NOT match a naive DFS of the visible tree** - confirmed real order when checking "Home": `home, desktop, documents, downloads, notes, commands, workspace, office, wordFile, excelFile, react, angular, veu, public, private, classified, general` (documents/downloads before their own deeper descendants notes/commands/workspace/etc. are listed). Assert against this real order rather than assuming id order matches tree depth-first order.

See [[repo_traning_conventions]] for how this was implemented (`page-objects/checkBox.ts`, methods `treeItem`, `expandNode`, `toggleCheckbox`, `expandFullTree`, `expectNodeChecked/Unchecked/Indeterminate`, `expectResultItems`, `expectResultAbsent`).
